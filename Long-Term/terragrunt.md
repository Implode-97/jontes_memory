# Terragrunt And IaC Memory

## Data Platform Terragrunt Catalog

- For `data-platform-terragrunt-catalog`, prefer one shared real E2E lane in CI instead of multiple per-stack real deploy jobs.
- Root test flow should own mandatory setup and cleanup: copy the example tree, generate stacks, run root apply, defer destroy, and clean stacks.
- Preserve feature-based E2E stages, with names such as `team_identities_assert`, `workspace_assert`, and `catalog_sandbox_assert`, but avoid importing broad Terratest packages when only simple opt-in/out stage gating is needed.
- For the current CAV-132831 shared E2E scope, `wavebreakers` is the canonical team name and derived values should align to it, including `cc-wavebreakers`. Keep workspace assignment units/jobs, account budget policy units, old stack tests/jobs, and live mock locals out unless the user explicitly expands scope; the migration target is the new shared `platform-e2e-test`.
- Treat local real apply as optional debug work, not the promised default V1 workflow. Keep fast guard checks outside the real deploy lane and require explicit CI mutual exclusion for shared real deploys.
- Treat TOOCAN work in Terraform/Terragrunt Data Platform IaC as a product or platform-owner decision gate before implementation. Current indexed TOOCAN guidance is CDK golden-path centered around standardized constructs, CDK CI, cdk-nag/AwsSolutions checks, GitLab security policies, CI/CD security checks, and dependency updates; no explicit Terraform/Terragrunt acceptance path was found. Build a control-equivalence gap map and get TOOCAN/support sign-off before adding platform code.
- Cleanup reliability is a first-order design concern for shared Databricks Terragrunt E2E tests. Try root `terragrunt stack run destroy` first, then fall back to reverse-order generated-unit destroys when the root graph is broken. Keep Terraform-owned janitors as the primary cleanup mechanism, and reserve direct API force cleanup for protected/manual recovery paths. Preserve the copied `example/non-prod-env-e2e` tree when cleanup fails so local state and generated unit files remain available for manual reruns.
- Before rerunning shared catalog E2E after a failed cleanup, preflight leaked `sandbox_*` Unity Catalog resources such as catalogs, storage credentials, and external locations. The CI example has commented remote state, so local Terraform state can disappear with the job workspace while stable names such as `sandbox_firecrackers` remain; clean leaks with a principal that can manage the metastore or catalog before treating the next run as a code retry.
- Add explicit cleanup targets for new E2E features, and add fast guard or pre-hook tests before real apply. Guard coverage should catch lifecycle hook asymmetry between `plan`/`apply` and `destroy`, dependency outputs needed during destroy-time provider rendering, mock allow-lists that hide real destroy inputs, and output-driven prior-state row removals.
- Terraform output assertions are useful, but higher confidence comes from Databricks API checks that resources really exist and were removed.
- Keep the canonical example root aligned with currently supported regional infrastructure. Moving a downstream feature unit, such as sandbox catalogs, to a supported region is not enough if the root stack still deploys an unsupported regional workspace; either provision the missing shared prerequisites or trim the unsupported region until it can be added back deliberately.
- In `data-platform-terragrunt-catalog`, centralize the Databricks deployer service-principal application/client ID in `example/non-prod-env/_common_vars.hcl` as `databricks_deployer_service_principal_application_id = get_env("DATABRICKS_CLIENT_ID")`. Feature units such as `dbx_metastore` and `dbx_workspace` should pass `local.common.databricks_deployer_service_principal_application_id` to wrapper inputs.
- When validating generated stack units, inspect the rendered `.terragrunt-stack` units under `example/non-prod-env/...`; raw `units/*` templates can fail outside their parent root/common file context.
- Do not resolve sibling unit outputs by scraping generated-unit `.terragrunt-cache` directories during the same `terragrunt stack run apply`. Terragrunt renders unit configs before downstream sibling caches or outputs may exist, so prefer normal dependency contracts, staged stack design, or caller-provided normalized inputs.
- In explicit Terragrunt stack-generated units, direct `dependency.*.outputs` interpolation may still be unavailable during generated HCL evaluation even on Terragrunt v1.0.8. For CAV-131446 `dbx_compute_policy`, CI showed `dbx_workspace_assignments` applying successfully, then `dbx_compute_policy` failing with `There is no variable named "dependency"` at direct dependency references; treat that as an evaluation-scope issue, not a missing upstream output.
- For CAV-131446 default Databricks compute policy grants, derive team eligibility directly from active rows in `workspace_teams.yml` and enabled teams in `_org/teams.yml`, then generate the canonical principal `grp_team_<team_key>_contributor`. Keep `dbx_workspace_assignments` as an ordering-only dependency for default grants, and avoid reintroducing workspace-assignment output reads into this generated-stack path unless the stack/autoinclude model is deliberately changed and proven.
- For CAV-131446 catalog work, MR `!29` is the June 2026 integration MR for the compute policy forward-port from Santosh's branch, adapted onto the modular E2E baseline. The catalog-side unit, examples, guard tests, and docs were ported; the separate modules branch that adds `compute_policy_team_assignments` was not part of the catalog port unless explicitly pulled in later. Recheck GitLab before using exact branch heads because these facts are a dated snapshot.
- For compute policy unit simplification, normalize automatic defaults and explicit requests into minimal seed maps first, guard duplicate keys before collapse, and project once into the module `policy_instances` shape. Keep product execution policy generation separate until V1 scope changes. Prefer native Terragrunt dependency blocks over shell/JQ output readers when the evaluation context supports them; for live commands, mocks should not hide missing required upstream data.
- For CAV-131446 compute policy profiles, use `_org/cluster_policy_profiles.yml` with root `compute_profiles`; profiles are file-backed via `definition_file`, not inline `policy_definition` or `policy_definition_json`. `is_default: true` marks automatic low-risk defaults, while explicit team opt-ins live on enabled `_org/teams.yml` rows as `compute_profile_ids`. Keep workspace-local `cluster_policies.yml` out of team sandbox policies and reserve it only for optional product execution policy requests unless the model deliberately changes.
- For generated team compute policies, propagate team `cost_center` and `unit_name` into the module `authoritative_custom_tags` so hidden `custom_tags.*` rules preserve cost attribution. Keep profile IDs honest to their cost and blast radius, and keep auto-termination lower bounds above zero.
- For catalog MR !21 / CAV-132831 continuation, the corrected registry pins from modules pipeline `7662011` are `databricks-account-metastore-wrapper/databricks` `0.4.0`, `databricks-account-metastore/databricks` `0.4.0`, `aws-databricks-workspace-wrapper/aws` `0.5.0`, and `aws-databricks-workspace/aws` `0.5.0`; branch commit `fe3f660` updated `dbx_metastore` and `dbx_workspace` to those wrapper versions.
- The `dbx_sandbox_catalogs` apply-only `before_hook` controlled by `DBX_SANDBOX_PREREQ_WAIT_SECONDS` is a diagnostic and stabilization probe for fresh workspace/metastore propagation, not permanent architecture. The original 900-second probe has been narrowed in the modularization follow-up; keep the passing fixed wait in that refactor, but implement bounded readiness polling as a separate deliberate change before workspace-level UC consumers.
- For `dbx_sandbox_catalogs`, preserve lifecycle guard JSON and tombstone handling as safety controls. Narrow cleanup is preferred: reuse active sandbox locals, split dense sandbox-definition construction into named locals, document or remove surprising init hooks such as `rm -rf tests/`, and explain order-only team identity dependencies.
- In Terragrunt catalog reviews, accept thin `_account.hcl` selector files and explicit one-unit `terragrunt.stack.hcl` files when they preserve catalog/live symmetry, keep generated unit paths visible, and make folder-level target edits obvious. Penalize safety-looking metadata that is not consumed or clearly descriptive, duplicate target/key identity fields, and commented placeholder units that imply committed future scope.
- For provider-parent reviews, first prove active consumers by search before judging line count. Preserve the active Databricks account provider and AWS assume-role provider contracts. Penalize dormant simple AWS or Databricks workspace provider parents, duplicate generated provider ownership, hard-coded mock workspace URLs, and alternate provider modes without a current unit contract.
- Workspace provider selectors are high-impact even when tiny. Reward self-targeting selectors whose folder name, `workspace_target_name`, dependency path, mock URL, active stack unit, and colocated YAML align. Cross-workspace selectors or live dbcore selectors must make the generated-unit dependency path explicit, usually through `.terragrunt-stack/dbx_workspace`; `workspace_target_name` alone does not prove the dependency path is safe.
- Treat orphan example roots, broad commented backend templates, and unused root locals as cleanup candidates unless an active generated stack, test fixture, or documented path consumes them. Active root S3 remote state with `use_lockfile = true` is a good KISS pattern; stale DynamoDB lock-table comments or dormant backend blocks are drift.
- For shared Terragrunt E2E, keep the main proof path clear: setup, apply, active assertions, cleanup. Root destroy plus reverse-order generated-unit janitors and preserving the copied tree on failure are justified recovery machinery, but cache/state scanning, broad diagnostics, and commented lifecycle matrices should live behind clearly named recovery helpers or be removed until active.
- Fast Terragrunt guard tests should execute the real generated-unit and shell-hook boundary while using narrow fake `terraform output -json` fixtures. Prefer table-driven lifecycle matrices, small case runners, narrow helper return values, and centralized Terragrunt env/options setup over repeated row setup or mini-framework env merging.
- Paired Terragrunt guards over the same generated JSON should make validation ownership explicit. Duplicated jq validation is acceptable only as a conscious fail-closed choice for independently runnable scripts; otherwise prefer one tiny shared validator or a single hook ordering. Missing, malformed, or renamed Terraform outputs must not silently become first-apply bootstrap state.
- The `dbx_workspace` catalog unit is active and generally worth preserving: keep live wiring in Terragrunt, delegate detailed AWS/Databricks behavior to the wrapper, and preserve the aliased Databricks MWS provider contract. Cleanup should target repeated registry-token/test-removal boilerplate, redundant default providers, and broad fixed readiness sleeps in favor of bounded readiness checks once workspace-level Unity Catalog consumers are stabilized.
- For MR !26 / `feature/codex/CAV-132831-platform-e2e-modularize-tests`, compare against `origin/master`, not the old CAV-132831 feature branch. Preserve it as a refactor-only E2E file split: no readiness polling, no sandbox AWS diagnostics, no apply retry wrapper, and no revived investigation helpers unless they directly support a later polling change. The review stance is accept with narrow cleanup, not broad rework.
- For MR !26 shared E2E output assertions, keep the normal assertion path on `terragrunt output -json` for readability and interface consistency. Prefer Terratest's `terragrunt.OutputJSONContextE` or equivalent library output helpers over hand-rolled shell output where possible. If Terragrunt returns `EOF` or similar on a generated unit, direct `.terragrunt-cache` or local `terraform.tfstate` probing is acceptable only in explicitly named recovery helpers; keep cache/state scanning out of stage assertion files and document fallback evidence in CI trace review.
- For E2E Terragrunt output decoding, prefer a hybrid shape: read the top-level output JSON into a `map[string]json.RawMessage` envelope and decode only asserted keys through a small typed helper such as `requireOutputValue[T]`. Keep nested structs for domain objects that are asserted heavily, such as sandbox catalog or grant objects, but avoid broad top-level mirror structs and fully dynamic `map[string]any` assertions.
- In modular E2E stages, each stage should assert the feature it owns. Put metastore output contract checks in a dedicated `metastore_assert` stage; downstream workspace or sandbox stages may read upstream outputs as fixture values but should not repackage or re-test broad upstream output bundles.
- After the sandbox catalog wrapper package rename, catalog units should use `aws-databricks-account-catalog-wrapper-sandbox/aws`, not the old `aws-databricks-catalog-wrapper-sandbox/aws`. Renamed GitLab Terraform module packages can reset to version `0.1.0`, so verify the exact package name and version in the registry instead of assuming monotonic versions across renames.

## Data Product Catalogs

- Keep Databricks data product catalogs as a separate public feature from sandbox catalogs. Prefer a new regional `dbcore` unit such as `units/dbx_data_product_catalogs` with its own Terraform wrapper and YAML registry, not product modes inside the sandbox catalog wrapper.
- The V1 regional input file is `data_products.yml`, with each product row containing an explicit `env_cycles` map. Product-level `compute_profile_ids` and provider-neutral raw `oidc_federation` entries should copy to each env-cycle service principal.
- Each active data product env-cycle should create one service principal, federation policies, bucket, external location, and catalog. Use sandbox patterns for storage pools, lifecycle guards, and tests where the analogy is clean, but keep workspace assignment and catalog workspace bindings out of this feature until a sibling feature owns them.
- Data product lifecycle states are `enabled`, `destroy_pending`, and `destroyed`; row removal is guarded until a destroyed apply. `break_glass_enabled` lives per env cycle, defaults true except for `prd`, and grants `grp_team_<owner_team_key>_owner`.

## Live Terragrunt Reviews

- Treat live `_common_vars.hcl` as a contract with generated units and provider parents, not a constants bag. Preserve stable enterprise facts such as naming prefix, backend bucket/region, Databricks account facts, and default tags; check every active consumer before treating a missing local or stale tag as low-risk cleanup.
- Live `_account.hcl` selectors should stay explicit: select the canonical target key such as `dbcore_euc1`, then pass through only the narrow locals consumed by generated units and provider parents. Prefer explicit target keys over path-derived magic; penalize wrong regional keys, unused alternate modes, and workspace-only fields leaking into account selectors.
- Live provider parents score well when they have visible consumers and one clear generated provider contract. Dormant alternate provider modes, ambient-credential parents without an explicit use case, or files sharing the same generated output path as an active parent should be treated as cleanup.
- Live Terragrunt target runners should stay thin around `terragrunt stack run`; current `stack run` already generates the selected stack. Extra target `stack generate` calls, broad sibling generation loops, or filesystem-derived fan-out need a documented stale-output or dependency reason and a visible blast-radius boundary.

## Live YAML Inputs

- Live Terragrunt YAML files should describe platform intent in flat, readable structures rather than deeply nested documents.
- Use `_org/teams.yml` for stable team metadata, optional role override files only for deviations, region or metastore scoped files for sandbox and data product catalogs, and workspace scoped files for bindings such as policies.
- Preserve the difference between "intentionally empty" and "malformed or missing." Do not silently normalize a missing required top-level key to an empty list.
- Live registry-looking YAML must match active consumer contracts exactly. Verify filename, root key, and row schema before treating team, workspace, sandbox, budget, or lifecycle data as supported; delete dormant registries or align them fully before adding future-looking entries.

## Terratest Usage

- For Terratest Terragrunt work, verify against current upstream source when deprecations or best-practice drift are in question instead of relying on older docs.
- Terratest `v1.0.0` uses `StackGenerateContext`, `StackRunContext`, and `StackCleanContext`; upgrading to it raises the Go toolchain requirement to `1.26`, which must be reflected in CI before the upgrade is complete.
- Avoid `github.com/gruntwork-io/terratest/modules/test-structure` for simple stage gating in this repo. In Terratest `v1.0.0`, it compiles save/load helpers that import heavy optional dependencies such as AWS, Kubernetes, Packer, SSH, OPA, and Terraform. Prefer a tiny local helper unless those save/load helpers are genuinely needed.

## Serverless Budget Policy Reviews

- The serverless budget policy feature moved from workspace-scoped to account-wide/account-scoped. Do not treat account scope or empty workspace bindings as automatically wrong.
- Still flag stale Jira, docs, or ADR text if it describes a workspace-scoped contract after the implementation has moved to account-level Databricks serverless usage policy semantics.
- For `dbx_budget_policy` or successor `serverless-budget-policy` work, review account-level naming collision risk, cost attribution tags, and explicit E2E or CI rollout decisions.
- For account-scoped Databricks controls, `workspace_ids = []` or empty workspace bindings are explicit all-workspaces activation decisions, not harmless defaults. Avoid automatic enablement from shared registries unless blast radius and opt-in semantics are named; platform cost/owner attribution tags should not be casually overridden by later `extra_tags` merges.

## Generated Files And Ignore Rules

- Terragrunt catalog `.gitignore` should track actual generated artifact names from CI and scripts. Prefer narrow rules for Terraform state/plans, `.terragrunt-cache`, `.terragrunt-stack`, copied E2E environments, guard JSON, Python caches, and Terratest caches; remove stale names such as `changed-modules.txt` when detect jobs emit `changed-units.txt`, `bump-type.txt`, `unit-versions.json`, or `unit-versions.txt`.
- Treat broad `.terraform.lock.hcl` ignores as suspicious unless the repo clearly treats lock files as local Terragrunt artifacts.

## Review Workflow

- The `ng-iac` workspace contains a repo-local `skills/iac-mr-review` skill for reviewing colleague MRs in Data Platform IaC repos.
- For future catalog, module, and live MR reviews, combine `iac-mr-review` with `iac-code-style`, fetch or inspect the MR branch, and read the related Jira story, Need field, or ADR before accepting scope changes.
