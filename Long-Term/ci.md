# CI Preferences

## Terraform Module Repositories

- In older Terraform module CI, prefer detecting changed modules and generating targeted dynamic jobs for those modules plus modules that depend on them, instead of running every module on every change.
- In `data-platform-terraform-modules`, the dynamic module test runner runs plain `terraform test` for each selected module, so every `tests/*.tftest.hcl` file in that module is included in normal MR CI. Wrapper `real_tests/` are not covered unless wired into a separate job. Keep live apply tests out of `tests/` unless they should run in ordinary MR CI, and use protected, manual, or scheduled CI for live wrapper smoke tests that need shared Databricks resources.
- For non-master Terraform module publishing, preserve SemVer-aware release candidates when that repository still uses this pattern: the first non-master publish can start at `0.0.0-rc.1`, and unclear branch types can fall back to a `bugfix` bump.
- In `data-platform-terraform-modules`, publish-time source rewriting in `ci/scripts/deploy-changed-modules/rewrite_publish_module_sources.py` should resolve unchanged local dependency modules by exact Terraform registry identity `<module>/<system>`, not by bare module name. GitLab package lookup by `package_name=<module>` can return similarly prefixed wrapper packages and generate child module version pins that do not exist.
- Publish-time module source rewriting must fail if a changed intra-repo dependency is missing from the same pipeline's computed versions. Falling back to the registry latest stable for a changed dependency can publish stale dependency pins.
- Terraform module release helpers should treat tag history and breaking-change markers as safety-critical: hard-fail if `git fetch --tags` cannot refresh the release source of truth, scan full commit messages for `BREAKING CHANGE` footers, and validate all package/version inputs before uploading immutable registry artifacts.
- When republishing Terraform wrappers after dependency lookup fixes, README-only touches to wrappers and direct child modules can be intentional version triggers rather than Terraform behavior changes. Preserve those touches unless the release strategy changes so wrappers can pin freshly published child module versions.
- If a CI test command excludes packages, such as `go test $(go list ./... | grep -v ...)`, verify the excluded package has an active replacement lane included by root CI. An unused job file in `ci/` is not coverage.
- Check current repository state before applying these historical CI preferences; they came from earlier module-repository work and may have evolved.

## Data Platform Terragrunt Catalog

- In `data-platform-terragrunt-catalog`, Codex MR review must stay asynchronous relative to `fmt`, `guard-tests`, and `platform-e2e-test`. A small detect-stage gate such as `print-changed-units` is acceptable, but Codex review should not block the real test lane.
- Catalog Codex review jobs may live in a later `review` stage with `needs: []` so they start immediately and remain non-authoritative. Keep README and workflow comments aligned with that shape, including intentional MR/push/web pipeline behavior and any duplicate branch/MR pipeline policy.
- The shared-environment E2E lane is the authoritative serialized CI lane for real deploy validation.
- Live E2E CI jobs need credential lifetime beyond the nominal test timeout so deferred cleanup can still authenticate. Prefer structured STS/OIDC parsing, explicit token preflight, and small checked-in bootstrap helpers over dense shell embedded in YAML.
- Before pushing or rerunning the shared E2E lane after a failed apply or cleanup, do a cleanup pre-flight: inspect the prior failed trace, cleanup or janitor result, and likely leftover Databricks, AWS, or Terragrunt-managed resources. Call out unverified orphan risk instead of blindly starting another long pipeline.
- Keep Terraform-owned cleanup as the normal shared E2E path: root Terragrunt destroy first, then reverse-order generated-unit Terraform janitors from cached state. Put API force-deletes behind separate protected/manual reset jobs or scripts with dry-run output and deterministic test-resource filters, not in normal CI cleanup.
- The fast guard-test lane uses Terragrunt-generated hook scripts that call `jq`. In the `alpine/terragrunt` CI image, install `jq` with `apk add --no-cache` whenever creating or recreating the guard-test job, or the guard suites fail before useful assertions run.
- For Terratest and Terragrunt failure investigations, fetch the real GitLab job trace when it is accessible. Pasted log excerpts are fallback evidence only.
- Save the raw trace and run the repo-local `skills/ci-log-triage/scripts/filter_ci_log.py` before diagnosing. Start from the first retained failure block; later cleanup errors are often fallout from the earliest contract, permission, or apply failure.
- For `platform-e2e-test` failures during Terraform init in generated `dbx_metastore` or `dbx_workspace` units, check GitLab Terraform module registry publication state and generated wrapper package tarballs before debugging Databricks auth or cleanup fallout. Missing or wrong child module versions can surface as `Unresolvable module version constraint`.
- For catalog MR !21 / CAV-132831 E2E retries, keep separating first apply failure from cleanup fallout. If `dbx_metastore` fails creating Databricks account group `platform-tratondev-dmp-euc1-metastore-admins` because it already exists, clean up or import that orphaned group before rerunning; janitor destroys may still look incomplete when workspace or metastore dependency outputs are unavailable during destroy.
- For sandbox catalog UC S3 or external-location READ failures, classify the failure before changing naming, sleep duration, or policy shape: Databricks cannot assume the role, Databricks can assume but S3 denies read/list, the storage credential has the wrong role ARN or external ID, file-events validation is incomplete, fresh workspace/metastore assignment is not ready, or Terraform generated a policy from wrong inputs.
- Sandbox catalog external-location READ failures right after IAM policy or trust creation, or even after a fixed 30-second wait, can still be AWS/Databricks propagation when the plan has the expected `credential_name`, IAM role policy, UC trust external ID, and no blocking bucket policy. Verify IAM state read-only with IAM simulation, CloudTrail STS/S3 evidence, bucket policy, storage credential state/API fields, and Databricks storage-credential validation output when possible; then prefer readiness or retry around Databricks validation over changing the bucket or role permission contract.
- Do not treat a passing sandbox wrapper module real test as proof that the Terragrunt live E2E topology is fixed. The module fixture can use a pre-existing eu-west-1 workspace/metastore and randomized temporary names, while catalog E2E uses a freshly created eu-central-1 workspace/metastore plus deterministic names such as `tratondev-dmp-euc1-sandbox-uc-role`, `sandbox_tratondev_dmp_euc1_storage`, and `tratondev-dmp-euc1-sandbox-firecrackers`.
- If CAV-132831 E2E only passes after `DBX_SANDBOX_PREREQ_WAIT_SECONDS=900`, treat that as evidence toward fresh workspace/metastore Unity Catalog readiness, not proof that a fixed sleep is the final design. Prefer a bounded readiness check before sandbox catalog apply.

## Data Platform Terragrunt Live

- Live root CI scores well as a compact include index with explicit workflow rules, global Terraform/Terragrunt automation defaults, and visible parent stages for bootstrap, validation, target preparation, and plan triggering. Score down inactive or undocumented stages, duplicate MR/branch pipeline creation rules, root defaults without consumers, and docs that lag current child-pipeline behavior.
- Live target catalogs can stay flat and active-scope-only when `stack_dir`, `display_name`, and `kind` make generated CI labels and job variables readable. Prefer one canonical target/account key, or a documented composite key, over repeated shorthand such as `target: dbcore` across regions; derive duplicate region facts unless a real distinction exists.
- Target-aware live pipeline generation should be boring and explicit. Use one tested path-matching model, one structured target parser, explicit dependency declarations or actual dependency-path parsing, fail-closed no-match behavior, and current GitLab child-pipeline status mirroring where available.
- Generated child-pipeline templates are acceptable when they stay to a small plan gate, manual deploy gate, and resource locking. Make parent-supplied variables such as `AWS_REGION`, `STACK_DIR`, `STACK_LABEL`, and `TARGET_RESOURCE_GROUP` discoverable near consumers; avoid tribal-knowledge variable provenance and stale `strategy: depend` wiring.
- GitLab HTTPS clone authentication and Terraform registry authentication are separate concerns in these Terragrunt CI jobs. Preserve the `gitlab-ci-token:${CI_JOB_TOKEN}` clone rewrite and Terraform `TF_TOKEN_<encoded-host>` convention when both source types are active, but document host encoding, token precedence, and any hard-coded enterprise GitLab host instead of adding broad configurability.
- Live validate jobs should make the actual command obvious, such as `terragrunt hcl fmt --check` or one stack generate/init helper. Preserve explicit `changes` gates and final `when: never` fallbacks, but collapse duplicated branch/source rules when behavior is identical and question auth-heavy base templates when generation/init does not clearly require credentials.

## GitLab CI Review Heuristics

- For MR reviews, compare `glab mr view` SHA, source branch API, MR commits, and MR pipelines before treating the newest detached pipeline as the current reviewable head. A failed or rolled-back pipeline can remain visible after the branch points elsewhere.
- For dynamic child pipeline gates, prefer `trigger: strategy: mirror` over stale `depend` semantics when a trigger job should mirror downstream status. Set `needs.artifacts` only when the bridge job itself consumes artifacts.
- CI helpers that read or write GitLab APIs should handle pagination, set network timeouts, bound log output, and use explicit write-capable tokens for MR notes. Do not assume `CI_JOB_TOKEN` can create notes unless the instance explicitly supports that write path.
- AI/Codex review wrappers that bypass sandboxing need least-privilege runtime boundaries: pinned tooling, narrow environment allowlists, isolated GitLab read/write tokens in tiny helpers, cleanup on launch failure, and explicit acceptance of any external sandbox precondition.
- Release lanes should serialize persistent release work with `resource_group`, guard against tag-pipeline recursion, make protected/default-branch assumptions visible, and state manual `allow_failure` intent.
