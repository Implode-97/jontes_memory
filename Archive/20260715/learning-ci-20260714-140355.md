# Learning: MR 32 green pipelines still have failed root destroys

## Context
Comparing `data-platform-terragrunt-catalog` MR !32 and master platform E2E traces clarified whether the workspace root-bucket failure is intermittent.

## Learning
At the current 10-unit MR topology, pipelines `7968852`, `7969323`, and `7969847` all failed the canonical root Terragrunt destroy with `BucketNotEmpty` on the dev workspace root bucket. They were green only because janitor recovery emptied the bucket and completed cleanup. By contrast, master pipelines `7954622` and `7961525` used nine units without the SQL warehouse and completed root destroy normally. Thus pipeline status must not be used as evidence that root destroy succeeded. The combined topology consistently reproduces the condition in available runs, but the traces do not record object keys or the writer, so they cannot prove whether SQL warehouse creation, another workspace operation, background Databricks activity, or timing produced the object.

## Future Use
Inspect root-destroy logs separately from job status. Log bucket keys and timestamps before destroy, and use S3 data events when writer attribution matters.
