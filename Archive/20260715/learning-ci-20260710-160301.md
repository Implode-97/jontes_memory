# Learning: Empty remote state is not first-apply absence

## Context
GitLab master pipeline `7961176`, job `36206011`, failed on the first `dbx_external_locations` apply after S3 remote state was enabled. The lifecycle guard saw `terraform state pull` return JSON while the lifecycle output was absent, and incorrectly classified the unit as previously applied.

## Learning
An initialized or previously destroyed S3 backend can expose a valid empty Terraform state snapshot with lineage and serial metadata but no tracked objects or outputs. Testing only whether `terraform state pull` returns a non-empty object makes first apply fail. The job’s root cleanup later destroyed every created unit successfully, while external locations reported zero resources destroyed, confirming that cleanup was fallout-free and the external-locations state was empty.

## Future Use
Lifecycle guards should distinguish an empty snapshot from tracked infrastructure by using `terraform state list` or inspecting the state resource collection. Allow bootstrap when no addresses exist; fail closed when addresses exist but the required lifecycle output is missing, and on backend/auth errors. Test all three cases.
