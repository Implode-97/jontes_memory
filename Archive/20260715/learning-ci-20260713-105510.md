# Learning: MR 32 cleanup failure is integration-specific

## Context
Comparing raw E2E traces clarified why `data-platform-terragrunt-catalog` master `7fc05e3` succeeds while MR !32 fails after rebasing onto it.

## Learning
Master pipelines `7954622` and `7961525` run nine units, including the external-location lifecycle matrix, and destroy all nine successfully. The old MR head `226605f` also passed with nine units on base `7251565`, where it added the SQL warehouse but did not yet include external locations. The current MR combines both feature sets into ten units; its body succeeds, but workspace root-bucket cleanup fails. Therefore neither feature independently proved sufficient to reproduce the failure. The combined E2E topology exposes the non-empty bucket condition, and the branch's fallback cannot recover because `root_bucket_name` is not an output.

## Future Use
Describe this as a target-base integration/cleanup bug, not a master regression, stale pipeline, rebase corruption, or SQL-warehouse-only failure. Inspect bucket contents before claiming which resource writes them.
