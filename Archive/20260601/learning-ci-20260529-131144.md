# Learning: Check Shared E2E Cleanup Before Reruns

## Context
During the Data Platform Terragrunt catalog MR work, Codex pushed a new CI run after a failed shared E2E pipeline without first verifying whether the previous failure had left Databricks, AWS, or Terragrunt-managed resources behind.

## Learning
For shared real E2E environments, a failed cleanup is not just secondary noise. Orphaned catalogs, storage credentials, groups, IAM roles, buckets, or metastore attachments can make the next CI run fail after many minutes and can obscure the real bug. Before pushing or rerunning CI for this lane, inspect the previous failed job and cleanup result, identify possible leftover resources from the first failure, and only proceed when the environment is plausibly clean or the user has explicitly accepted the risk.

## Future Use
Before triggering or pushing another shared E2E run, perform a cleanup pre-flight: review the latest failed trace, check cleanup/janitor output, and call out any unverified leftover-resource risk to the user instead of blindly starting another pipeline.
