# Learning: Keep Terraform Janitor Primary for Shared E2E Cleanup

## Context
During the Data Platform Terragrunt catalog MR cleanup discussion, the user considered replacing the reverse-order E2E cleanup logic with manual or API force-deletes for Databricks and AWS resources.

## Learning
For the shared real E2E lane, keep Terraform-owned cleanup as the default path: root Terragrunt destroy first, then reverse-order generated-unit Terraform janitor from cached state. Explicit API cleanup is better as a separate protected/manual reset path because making it the normal CI cleanup would require custom dependency ordering, idempotency, retries, provider/account scope handling, and safety checks for metastores, workspaces, groups, S3 buckets, security groups, IAM policies, IAM roles, catalogs, storage credentials, and external locations.

## Future Use
When improving E2E cleanup, prefer strengthening the Terraform janitor and adding read-only leftover preflight checks. Add API force cleanup only as a tightly scoped manual recovery script or job with dry-run output and deterministic test resource filters.
