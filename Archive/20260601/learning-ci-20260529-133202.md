# Learning: sandbox external-location validation can outlive 30s IAM wait

## Context
In data-platform-terragrunt-catalog pipeline 7702363 job 34629660, platform-e2e-test failed creating `sandbox_firecrackers_location` for `s3://tratondev-dmp-euc1-sandbox-firecrackers/`.

## Learning
The generated sandbox wrapper policy shape was correct: the shared role `tratondev-dmp-euc1-sandbox-uc-role` had bucket-level `s3:ListBucket`, `s3:GetBucketLocation`, and multipart list access for `arn:aws:s3:::tratondev-dmp-euc1-sandbox-*`, object-level `s3:GetObject` and write/delete multipart access for `arn:aws:s3:::tratondev-dmp-euc1-sandbox-*/*`, Databricks UC trust used external ID `47dc11d5-0e29-4a29-b9b0-9d3d3961e518`, and the bucket policy only denied insecure transport. The failure happened about 30 seconds after IAM role policy creation, so IAM/Databricks propagation remains the leading root cause, not missing S3 permissions.

## Future Use
For similar failures, verify IAM state read-only first, then prefer retry/readiness handling around Databricks external-location validation over changing the bucket or role permission contract.
