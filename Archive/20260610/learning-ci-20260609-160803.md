# Learning: live Terragrunt root pipeline readability

## Context
This came from scoring `data-platform-terragrunt-live/.gitlab-ci.yml` for readability, KISS, YAGNI, maintainability, and live deployment safety.

## Learning
The live root CI shape scores well when it stays a small include index with explicit workflow rules, global Terraform/Terragrunt automation defaults, and visible parent stages for bootstrap, validation, target preparation, and plan triggering. The recurring readability risks are not the include split itself, but drift and hidden contracts: README stage lists that omit manual bootstrap, root defaults that most jobs override, broad push workflow rules that may create duplicate MR/branch pipelines, and registry-token globals that rely on included shell scripts to explain their purpose. Current GitLab docs also prefer `trigger:strategy: mirror` over the older `depend` behavior in child-pipeline triggers.

## Future Use
When reviewing live Terragrunt CI, keep parent wiring compact and explicit. Score down inactive or undocumented stages, root defaults without a real consumer, duplicate pipeline creation rules, and generated child-pipeline behavior whose docs lag current GitLab recommendations.
