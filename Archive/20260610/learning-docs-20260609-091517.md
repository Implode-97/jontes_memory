# Learning: Keep CI/CD Docs As A Map

## Context
This came from multi-agent scoring of the catalog repo's `docs/CI_CD.md` while comparing the document to the actual GitLab CI configuration and adjacent docs.

## Learning
CI/CD documentation is most valuable when it is a compact map of stages, jobs, branch behavior, and release gates. It should not become a second full source of truth for E2E execution, OIDC setup, or release semantics already documented in `TESTING.md`, `OIDC_AUTH.md`, and `RELEASES.md`. When a review lane exists, especially a non-blocking Codex/MR review stage, it must appear in the stage map so readers are not misled about what the pipeline actually contains.

## Future Use
When reviewing or writing IaC repo docs, prefer short CI tables plus links to deeper operational docs. Check root `.gitlab-ci.yml` and included job files before accepting stage/job claims, and flag docs that omit included lanes or misstate required versus optional `needs`.
