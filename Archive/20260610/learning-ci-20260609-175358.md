# Learning: live stack generate init CI job readability

## Context
This came from reviewing `data-platform-terragrunt-live/ci/validate/stack_generate_init.yml` for GitLab CI YAML readability, KISS, YAGNI, maintainability, duplicated rules, anchors, and job size.

## Learning
The `stack_generate_init` validate job is useful as a non-destructive early gate, but its behavior is much smaller than its CI scaffolding. The same path anchor and five-rule matrix are duplicated from `hclfmt`; root workflow already limits pipeline sources to MR, push, and web; branch-specific push and web rules currently express identical outcomes. Inheriting `.terragrunt-auth-job` is only justified if `terragrunt stack generate` or `stack run init` really need AWS/Databricks OIDC and role assumption during init.

## Future Use
For future live Terragrunt CI YAML reviews, accept thin validate gates, but prefer shared validate rules or collapsed source rules when branch branches do not differ. Call out auth-heavy base inheritance when the script is just generation/init and credentials are not clearly required.
