# Learning: Separate Terratest modules need CI-backed boundaries

## Context
This came from scoring `data-platform-terragrunt-catalog/tests/terratest/e2e/go.mod` for readability, KISS, YAGNI, and maintainability.

## Learning
Separate Go modules for Terratest E2E and guard tests can be justified when CI proves they are separate operational lanes. In this repo, the E2E module is live/destructive and runs from `tests/terratest/e2e`, while guards run from `tests/terratest/guards`, so duplicated `go.mod` metadata is acceptable. Direct dependencies should match visible imports, while indirect requirements and `go.sum` should remain `go mod tidy` owned rather than manually trimmed for appearance.

## Future Use
When reviewing Terratest module files, do not score a module split down merely because the dependency sets look similar. First verify whether CI, live-cost risk, or package ownership requires separate execution. Still penalize opaque local module names such as `terrateste2e`; prefer clear controlled or path-like names when a module is touched.
