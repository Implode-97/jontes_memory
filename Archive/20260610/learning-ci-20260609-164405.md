# Learning: GitLab clone auth and Terraform registry auth split

## Context
This came from multi-agent scoring `data-platform-terragrunt-live/ci/scripts/setup_gitlab_clone_auth.sh` for shell readability, GitLab/Terraform CI auth correctness, and maintainability.

## Learning
For these Terragrunt CI jobs, GitLab HTTPS clone authentication and Terraform registry authentication are separate concerns and both can be justified. Git clone/source downloads use GitLab's `gitlab-ci-token:${CI_JOB_TOKEN}` pattern, while nested Terraform module registry access uses Terraform's `TF_TOKEN_<hostname>` environment credential convention with encoded hostnames. The readability debt is usually not the auth mechanisms themselves, but hidden hostname coupling, unexplained Terraform hostname encoding, and duplicate token defaults between root CI variables and helper fallbacks.

## Future Use
When reviewing CI auth helpers, preserve the clone-vs-registry split if both source types are active. Prefer one visible token source of truth or an explicitly documented override/fallback, a short comment around job-scoped global Git rewrites, and a clear mapping between the raw GitLab host and the encoded `TF_TOKEN_*` variable.
