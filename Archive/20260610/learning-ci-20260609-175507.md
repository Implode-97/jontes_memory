# Learning: live validate job rule matrices

## Context
This came from multi-agent scoring `data-platform-terragrunt-live/ci/validate/hclfmt.yml` and `ci/validate/stack_generate_init.yml` for GitLab CI readability, KISS, and live validation safety.

## Learning
The validate jobs are strongest when their actual command is obvious: a plain `terragrunt hcl fmt --check` or a single root stack generate/init script. The readability debt is mostly in repeated scheduling scaffolding: separate default-branch and non-default push rules with identical behavior, duplicate web manual rules, broad `ci/**/*` triggers that rerun unrelated checks, and inherited base templates that pull in AWS or Databricks auth beyond what the job clearly needs.

## Future Use
When reviewing GitLab validate jobs, preserve explicit `changes` gates and final `when: never` fallbacks, but prefer collapsed rules when behavior is identical. Check whether CI-path triggers are intentionally broad, and whether the inherited base matches the command's real needs. Use comments sparingly to explain intentionally heavy auth or intentionally broad self-validation.
