# Learning: generated child pipeline contract visibility

## Context
This came from reviewing `data-platform-terragrunt-live/ci/target_deploy_pipeline.yml` as a new platform maintainer for readability, KISS, YAGNI, maintainability, and generated-child-pipeline contract clarity.

## Learning
A tiny generated child-pipeline template can be acceptable when it has only one plan gate, one manual deploy gate, shared resource locking, and no speculative modes. The main maintainability debt is hidden variable provenance: values such as `AWS_REGION`, `AWS_DEFAULT_REGION`, `STACK_DIR`, `STACK_LABEL`, and `TARGET_RESOURCE_GROUP` are supplied by the parent target generator and forwarded by bridge jobs, so the child template should make that contract discoverable near the variable consumers or by using clearer shared templates.

## Future Use
When reviewing live Terragrunt CI templates, reward small explicit gates but score down templates that require tribal knowledge to find forwarded variables, duplicated deploy scripts instead of existing deploy templates, or stale downstream `strategy: depend` wiring where current GitLab supports `strategy: mirror`.
