# Learning: GitLab Clone Auth Helper Needs Explicit Contracts

## Context
This came from reviewing `data-platform-terragrunt-live/ci/scripts/setup_gitlab_clone_auth.sh` for new platform maintainer readability, KISS, YAGNI, maintainability, and live CI auth clarity.

## Learning
The helper's dual behavior is useful: Git HTTPS clone rewriting with `gitlab-ci-token:${CI_JOB_TOKEN}` supports nested GitLab module clones, while `TF_TOKEN_gitlab_collaborationlayer__traton_com` supports Terraform registry credentials for the enterprise GitLab host. Its main readability debt is that the two mechanisms, required versus optional variables, token precedence, and encoded hostname contract are implicit in three executable lines. The hard-coded enterprise host is acceptable as live wiring only if it is intentionally enterprise-specific and named/documented as such.

## Future Use
For future live CI auth reviews, preserve necessary Git clone and Terraform registry auth. Prefer small comments/constants that label the GitLab host, explain `TG_TF_REGISTRY_TOKEN` fallback to `CI_JOB_TOKEN`, and document Terraform environment variable hostname encoding instead of adding broad configurability or new auth abstractions.
