# Learning: Keep Destructive Lifecycle Docs Precise But Not Repetitive

## Context
While scoring the `dbx_sandbox_catalogs` README, multiple reviewers found the staged destroy contract valuable and accurate, but also noted that the same lifecycle model appeared as a table, diagram, operator flow, reject list, and input-flow diagram.

## Learning
For destructive Terragrunt lifecycle docs, the lifecycle table and operator checklist are usually the highest-value artifacts. Extra diagrams are justified only when they clarify a different contract. Repeating the same state machine in several forms increases drift risk. Coverage wording should separate local guard-policy tests from active shared E2E scope, especially when full lifecycle E2E is commented out or deferred.

## Future Use
When reviewing READMEs for lifecycle-managed IaC, keep staged destroy, tombstone, row-removal, and cleanup prerequisites explicit. Prefer one authoritative transition table plus concrete operator checkpoints. Add exact prerequisites and live coverage limits, and avoid vague phrases like “empty file” when implementation requires `key: []`.
