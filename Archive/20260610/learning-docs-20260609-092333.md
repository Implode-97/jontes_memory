# Learning: Keep Naming Docs As Stable Glossaries

## Context
This came from multi-agent scoring of the catalog repo's `docs/NAMING.md` against current Terragrunt units, example target names, and GitLab CI job names.

## Learning
Naming documentation is useful when it defines stable repo-local shorthand and terms that prevent churn, such as `dbx`, `dbw`, `uc`, `ws_dev`, `ws_prd`, and canonical target names. It becomes fragile when it embeds incomplete inventories of CI jobs or feature-specific contracts that are already owned by CI YAML, unit READMEs, or architecture docs. A file titled “Naming Conventions” should either contain real conventions or be framed as a repository glossary.

## Future Use
When reviewing IaC naming docs, keep stable cross-repo vocabulary and remove or link out volatile inventories. Add only high-signal tokens that a new maintainer would otherwise misinterpret, and avoid duplicating CI job lists unless the naming doc is explicitly responsible for them.
