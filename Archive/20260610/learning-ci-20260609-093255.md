# Learning: Release Docs Must Own Release Semantics

## Context
This came from scoring the catalog repo's `docs/RELEASES.md` against the actual detect and release CI scripts, SemVer behavior, and GitLab pipeline variables.

## Learning
A specialized release document should not be only a tiny trigger summary if release behavior creates persistent Git tags. It should own the durable release semantics: unit-scoped tag format, pipeline-source behavior, default-branch stable releases, non-default RC releases, bump precedence, active RC mismatch handling, and remote tag history as source of truth. README or CI overview pages can summarize this, but the release doc should be the compact authority.

## Future Use
When reviewing release docs, compare them to detect/release scripts and current CI YAML. Flag missing stable-vs-RC behavior, manual release intent, required versus optional gates, and tag annotation shape. Keep implementation internals out unless they affect persistent tag correctness or operator decisions.
