# Learning: Catalog E2E Workspace Root Bucket Cleanup

## Context
While triaging GitLab job 35747031 for `data-platform-terragrunt-catalog` MR !32, the platform E2E apply and assertions succeeded but cleanup failed during the `ws_dev` Databricks workspace destroy.

## Learning
The concrete cleanup blocker was `BucketNotEmpty` on the Databricks workspace root S3 bucket, not the new serverless SQL warehouse unit. The SQL warehouse destroyed cleanly before the failure. The workspace wrapper creates root storage through `aws-s3-bucket-secure`, and that hardened bucket module intentionally does not expose S3 `force_destroy`; callers are expected to empty buckets through an explicit cleanup path.

## Future Use
For future catalog E2E failures, treat root stack `EOF` errors as a possible Terragrunt wrapper symptom and search the raw trace for the first provider-level error. If the blocker is a root storage bucket, recommend explicit E2E cleanup or preflight bucket-emptying rather than adding implicit force deletion to the reusable secure bucket primitive.
