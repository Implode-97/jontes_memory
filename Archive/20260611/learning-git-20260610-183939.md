# Learning: Forward-port colleague MR work onto accepted E2E baseline

## Context
During the data-platform Terragrunt catalog E2E work, the user needed to continue `feature/najvoa/CAV-133921_e2e_testing_serverless_budget_policy` after the modular E2E branch became the intended baseline.

## Learning
When a large E2E refactor is the new best-practice base, the user prefers creating a new branch from that E2E MR branch and forward-porting the colleague branch's feature intent into the newer structure. Do not directly rebase or rewrite the colleague branch unless the user explicitly asks for that.

## Future Use
For similar follow-up MRs, first identify the accepted baseline branch, create a new integration branch from it, then adapt older feature changes to the current style and helper contracts. Keep the original branch untouched so the user can review and later MR the new branch back into the colleague branch.
