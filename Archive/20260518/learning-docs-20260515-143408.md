# Learning: keep documentation simplified but not over-pruned

## Context
While refactoring the Terragrunt catalog repo docs around the new shared E2E model, the user asked whether the documentation had been pruned too aggressively and requested a repair pass.

## Learning
The user wants duplicate explanations reduced, but still expects the key operational details to remain in the correct deep docs. Simplifying structure is good; stripping out troubleshooting, pipeline behavior, auth failure modes, or architecture wiring detail is not.

## Future Use
When rewriting repo docs for this user, keep a light top-level structure but restore dense operational detail in the specialized documents instead of collapsing them into summaries.
