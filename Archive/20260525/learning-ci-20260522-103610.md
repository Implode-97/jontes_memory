# Learning: module registry lookup matches wrapper names too broadly

## Context
While investigating the catalog E2E failure for `databricks-account-metastore-wrapper` version `0.3.0`, Codex traced where the generated child dependency version `databricks-account-metastore/databricks` `0.2.0` came from.

## Learning
The committed wrapper source only uses `source = "../databricks-account-metastore"`. During publish, `ci/scripts/deploy-changed-modules/rewrite_publish_module_sources.py` rewrites local sources to GitLab registry sources and inserts a version. For unchanged dependency modules, it calls the GitLab packages API with `package_name=<module>`. Querying `package_name=databricks-account-metastore` returns both `databricks-account-metastore/databricks` and `databricks-account-metastore-wrapper/databricks`, so the script can pick wrapper version `0.2.0` as if it were the child module's latest stable version.

## Future Use
When debugging generated Terraform module version pins, verify the package API query result set, not just the final tarball. The publish rewrite should filter exact package names such as `<module>/<system>` before selecting the latest stable dependency version.
