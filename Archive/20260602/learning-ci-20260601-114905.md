# Learning: Sandbox Wrapper Module Test Is Not A Full E2E Readiness Proof

## Context
While investigating Databricks external-location READ failures, the user pushed back that unique resource names are not the root cause or solution.

## Learning
For sandbox catalog READ failures, do not argue from the passing modules repo real apply test alone. That test runs the wrapper against a long-lived shared eu-west-1 workspace/metastore fixture and only proves the wrapper can work in isolation. The Terragrunt E2E creates the metastore, workspace, workspace admin assignment, storage credential, IAM trust/policy, and external location in one full bootstrap flow. A pass in the module fixture does not exercise fresh Databricks account/workspace/metastore readiness propagation, CI deployer permission propagation, or the exact live topology.

## Future Use
When diagnosing this class of failure, compare variables and policies, then seek proof from CloudTrail, IAM simulation, and Databricks storage-credential validation before changing naming or sleep duration.
