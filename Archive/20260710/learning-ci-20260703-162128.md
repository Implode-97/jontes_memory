# Learning: direct external-location apply reproduces EOF

## Context
GitLab pipeline `7918858`, job `35953170`, ran commit `ce1e2d7` with targeted generated-unit no-op diagnostics after the normal platform E2E apply and output assertions.

## Learning
The first direct generated-unit `terragrunt run -- apply` for `dbx_external_locations` failed with EOF before the dev workspace reapply probe started. The run retrieved the regional metastore output and dev workspace output successfully, then copied files into the external-location `.terragrunt-cache`, regenerated `provider_aws.tf`, and failed before `provider_databricks_workspace.tf`, `external_locations_guard.json`, `I0720`, `I0721`, or Terraform apply. This rules out the dev workspace reapply, root `run --all` orchestration, guard scripts, and Terraform/provider execution as required triggers for this failure.

## Future Use
Next diagnostics should compare direct Terraform cache `apply` versus Terragrunt single-unit apply for `dbx_external_locations`, or reduce the unit generate/dependency blocks to isolate the Terragrunt EOF boundary.
