# Learning: Serialized repeated root apply still fails before Terraform

## Context
While triaging GitLab `platform-e2e-test` job `35872224` for `data-platform-terragrunt-catalog`, the diagnostic second `terragrunt stack run apply` was run with `TG_PARALLELISM=1`, `TG_LOG_LEVEL=debug`, `TG_NO_COLOR=true`, and `TG_SUMMARY_PER_UNIT=true`.

## Learning
`TG_PARALLELISM=1` did not prevent the repeated-root-apply failure. The second apply succeeded for the upstream units and then failed three downstream generated units with bare `EOF`: `dbx_external_locations`, `dbx_sandbox_catalogs`, and `dbx_workspace_assignments`. Debug logs showed dependency outputs were cached successfully and generated provider files were overwritten in `.terragrunt-cache`, but Terragrunt marked each unit failed before any `terraform apply` invocation, provider output, guard-script message, or YAML lifecycle mutation.

## Future Use
Treat this failure mode as repeated Terragrunt stack execution/cache/run plumbing, not as a Databricks provider, guard script, state overwrite, external-location feature, or stack parallelism problem unless new logs show Terraform actually starts for the failed unit.
