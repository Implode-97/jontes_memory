# Learning: Terraform module CI favors targeted dynamic jobs

## Context
Early Terraform module repository sessions designed CI behavior for module testing, release candidates, and changed-module detection.

## Learning
The user preferred CI that detects which Terraform modules changed and generates dynamic jobs for those modules, including modules that depend on a changed module. This avoids running everything while still catching downstream breakage. For non-master publishing, the preferred release-candidate behavior evolved toward SemVer `rc` suffixes, with the first non-master publish starting at `0.0.0-rc.1` and branch type fallback treated as `bugfix` when the branch does not clearly specify a bump.

## Future Use
When working in the older Terraform modules CI, preserve targeted dynamic child pipelines and semver-aware release candidate handling. Check the current repository state before assuming these historical preferences are still active.
