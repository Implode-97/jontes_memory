# Learning: memory uses capture first and dream later

## Context
The user designed the Obsidian Codex memory system across several May 2026 sessions, then asked for this retroactive capture pass.

## Learning
The intended memory workflow is two-stage. Normal sessions should create concise short-term learning files only when a reusable learning appears. Separate dream runs consolidate short-term notes into compact long-term category files, archive processed notes, log decisions, and avoid creating noise when there is nothing to process. Interactive dream runs may ask for clarification on unresolved conflicts; scheduled runs should process non-conflicting notes and leave ambiguous items in Short-Term. The memory root is also a Git repo, and successful dream runs should create local commits.

## Future Use
Do not run dream consolidation automatically during ordinary work unless the user asks or an automation invokes it. Use recall-memory before relevant work and capture-memory after substantive reusable learnings.
