# Learning: ADRs should be polished markdown, not raw dumps

## Context
Several sessions asked Codex to draft or refine ADRs for team identities, sandbox input coupling, and platform plans.

## Learning
When the user asks for ADR or plan text, they usually want polished normal markdown in the chat or a deliberate `.md` file, not a raw markdown code block. ADRs should include the useful sections only, such as Context and Problem Statement, Decision, Consequences, Decision Drivers, Considered Options, Additional Information, and Notes. Implementation references should point to GitLab or stable project references rather than local-only paths when the output is meant for sharing. Definition of Done and Mermaid diagrams are welcome when they make the plan easier to review.

## Future Use
Before writing docs, infer whether the output is for a repo file, Jira, ADR review, or chat. Keep it human-readable, remove accidental local path leakage, and preserve enough implementation detail for future agents.
