# Learning: Measure twice before cutting IaC fixes

## Context
This came from a Databricks Terragrunt E2E fix where I first implemented a standalone metastore deployer privilege unit without fully comparing it against the existing metastore admin group and service-principal membership design.

## Learning
For this user, root-cause investigations must include alternative solution paths and a close reading of the current IaC lifecycle before choosing a fix. A technically valid solution can still be wrong if it ignores an existing ownership model, duplicates lifecycle concerns, or creates a new feature boundary where the current code already has the right place to extend. The better solution here was to add the IaC deployer service principal to the existing metastore admin group rather than creating a separate deployer-grant feature.

## Future Use
Before editing Terraform or Terragrunt, enumerate the plausible fixes, map them to existing modules, ownership groups, and lifecycle boundaries, then choose the option that best fits the current architecture. Measure twice, cut once.
