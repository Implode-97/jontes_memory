# Learning: Alias-dependent wrappers need static validate skips

## Context
The Terraform modules MR for Databricks external S3 locations failed in `terraform-static-checks` when the changed module set included `aws-databricks-workspace-catalog-wrapper-sandbox`.

## Learning
Plain standalone `terraform validate` is not meaningful for reusable wrapper modules that require caller-supplied aliased providers, such as `databricks.account`. The sandbox catalog wrapper can fail validation with `Provider configuration not present` even though its Terraform tests pass when providers are supplied through test files or live callers. CI should keep `fmt`, `init`, and TFLint coverage for these wrappers, but skip standalone validate in the generic static loop.

## Future Use
When a module MR touches an alias-dependent wrapper and static CI fails on missing provider configuration, inspect `ci/static-check.yml` before changing module contracts. Prefer adding the wrapper to the existing validate-skip condition and rely on module tests for provider-aware coverage.
