# Learning: Account membership README safety cue

## Context
During a read-only scoring review of `data-platform-terraform-modules/modules/databricks-account-group-membership/README.md`, the user asked for a Databricks account identity/domain perspective focused on readability, KISS, YAGNI, maintainability, and unsafe ambiguity.

## Learning
The README is mostly shippable when it shows the AWS accounts host, account ID, OIDC environment authentication, existing account identity lookups, and mocked test coverage. The main safety/readability gap is that account-scoped Databricks provider inheritance can be ambiguous in roots that also use a workspace provider. A README for this module should make the account-vs-workspace boundary explicit, ideally with an aliased account provider and module `providers` mapping, plus one sentence that this only manages account SCIM group membership and does not assign groups to workspaces, grant entitlements, or create missing identities.

## Future Use
For future Databricks account identity README reviews, preserve concise examples and OAuth/WIF guidance, but check provider alias clarity, account-only scope language, and drift between variable validation and input-table descriptions.
