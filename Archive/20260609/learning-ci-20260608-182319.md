# Learning: GitLab deploy jobs need explicit manual and trigger semantics

When scoring GitLab CI deploy jobs in the IaC repos, treat implicit manual-rule behavior as readability debt. If a job uses `when: manual` inside `rules`, make the intended blocking behavior explicit with `allow_failure` when the deploy controls persistent registry or environment changes.

For deploy jobs that depend on a trigger job before publishing, verify that the trigger job mirrors or otherwise represents downstream pipeline success. A deploy `needs` edge to a trigger job is only safety-preserving if the trigger status actually reflects child pipeline status.

Prefer `$CI_DEFAULT_BRANCH` over repeated hard-coded `master` when the repo/GitLab version supports it, and be precise about `rules:changes:compare_to` semantics for non-push or branch pipelines.
