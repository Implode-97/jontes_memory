# Learning: Treat Release Tags as Mandatory Input

## Context
While scoring `detect_target_version.py`, reviewers found that the script warned on `git fetch --tags` failure but still wrote version artifacts from local tags.

## Learning
If tag history is the declared source of truth for release version calculation, fetching tags must be a hard precondition before producing release artifacts. Warning and continuing can create stale `unit-versions.json`, and a later release job re-fetching tags does not fix already-computed versions. RC stream validation should also consider every active RC base, not only the latest-looking one.

## Future Use
When reviewing CI release-version scripts, penalize fallback-to-local-tags behavior unless there is a deliberate offline mode. Prefer a single parsed tag model per unit, typed JSON release artifacts, strict malformed-tag handling, and explicit validation that only one active RC base exists.
