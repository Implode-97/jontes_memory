# Learning: Verify MR head separately from latest pipeline

## Context
During a review of `data-platform-terragrunt-catalog` MR !22, GitLab reported a newer failed `refs/merge-requests/22/head` pipeline for commit `4e3625b`, while the MR source branch, MR `sha`, and MR commit list all still ended at `72d96c0`.

## Learning
For GitLab MR reviews, do not assume the newest MR pipeline SHA is the current reviewable branch head. A failed detached or rolled-back pipeline can remain visible as `pipeline` or latest build metadata even after the source branch points back to a different commit.

## Future Use
Before filing findings from CI traces, compare `glab mr view` `sha`, the source branch API, and `merge_requests/:iid/commits` against `merge_requests/:iid/pipelines`. Treat mismatched newer pipeline commits as validation ambiguity unless that SHA is still present in the MR commit list or source branch.
