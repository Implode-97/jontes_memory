# Learning: Keep Catalog Root READMEs as Maps

## Context
While scoring `data-platform-terragrunt-catalog/README.md`, multiple review agents found that the root README was useful but had drifted from the actual CI shape and duplicated deeper docs.

## Learning
For Terragrunt catalog repos, the root README should be a concise map: purpose, layout, current units, lifecycle-state pointers, short CI model, and links to deeper docs. Detailed pipeline, auth, release, E2E, troubleshooting, and local debug instructions should live in specialized docs. Any command examples that remain in the README must mirror the authoritative CI lane, including intentional exclusions such as the guard job omitting `dbx_workspace_assignments`.

## Future Use
When reviewing or writing catalog docs, penalize root READMEs that become a second source of truth for CI/testing details. Verify stage names and example commands against `.gitlab-ci.yml` and CI include files before treating README guidance as current.
