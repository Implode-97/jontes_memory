# Learning: Serialize and Guard Release Lanes

## Context
While scoring `data-platform-terragrunt-catalog/ci/release.yml`, the release wrapper was simple and readable, but reviewers found release-specific safety gaps that matter even when the YAML is small.

## Learning
Release jobs should have a higher explicitness bar than ordinary CI jobs. Even a thin wrapper around a tagging script should serialize concurrent release pipelines, explicitly deny tag-pipeline recursion when tags trigger pipelines, and make protected/default-branch assumptions visible in `rules`. Manual release jobs should also state their `allow_failure` intent consistently so maintainers can tell whether a manual release is optional UI noise or a blocking promotion step.

## Future Use
When reviewing GitLab release lanes, reward small wrappers and correct `needs:optional` usage, but look for `resource_group`, tag guards, protected/default-branch constraints, and explicit manual semantics before scoring them as production-ready.
