# Learning: Compute policy README consumer contract

## Context
During a read-only KISS/YAGNI/readability review of `data-platform-terraform-modules/modules/databricks-workspace-compute-policy/README.md`, the user wanted the perspective of a new platform maintainer and Terragrunt consumer.

## Learning
For this module's README, the important docs contract is not exhaustive validation detail. It is showing one minimal normalized `policy_instances` call, making clear that Terragrunt/catalog/live owns bundle expansion, workspace targeting, team mappings, and normalization, and showing how upstream workspace assignment data such as `workspace_team_assignments` becomes `acl_principals`. The README should also identify which outputs are stable downstream contracts, especially `policy_assignments` and policy IDs/names, versus broad diagnostic composites that may drift.

## Future Use
When reviewing or editing this module's docs, preserve the explicit mock-safe local test command and keep real-provider tests opt-in outside default `tests/` discovery. Prefer simplifying duplicated module-purpose and validation prose in favor of concrete consumer examples and output guidance.
