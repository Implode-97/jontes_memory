# Learning: catalog Codex review stage should be documented as asynchronous

## Context
During read-only scoring of `data-platform-terragrunt-catalog/.gitlab-ci.yml`, reviewers compared the root stage list, Codex review include, README pipeline section, and current GitLab `workflow:rules` and `needs: []` behavior.

## Learning
The catalog root CI file intentionally keeps Codex MR review separate from the real validation lane. The jobs live in a `review` stage but use `needs: []`, so GitLab starts them immediately even though `review` appears after `release` in the stage list. That shape is acceptable when Codex review must stay asynchronous and non-blocking, but README/docs should not describe it as part of the detect stage. The root workflow also deliberately allows MR, push, and web pipelines; if branch visibility is intentional, duplicate branch/MR pipeline objects should be called out.

## Future Use
When reviewing catalog CI, preserve asynchronous Codex review if requested, but check that docs, stage names, and workflow comments explain it. Watch for unintended tag push pipelines and for missing `$CI_OPEN_MERGE_REQUESTS` suppression if duplicate branch/MR pipelines are not intentional.
