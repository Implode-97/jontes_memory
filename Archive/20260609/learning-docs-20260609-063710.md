# Learning: Keep compute policy READMEs consumer-focused

## Context
During read-only scoring of `data-platform-terraform-modules/modules/databricks-workspace-compute-policy/README.md`, reviewers compared the docs against Databricks compute policy behavior, Terraform provider resources, and local IaC style.

## Learning
Compute policy module docs score well when they state the real boundaries: workspace-scoped Databricks provider, caller-prepared policy instances, module-owned deterministic names, hidden `custom_tags.*` injection, separate authoritative `CAN_USE` permissions, and catalog/live ownership of orchestration. The weak pattern is turning the README into implementation inventory or future planning: giant one-line object types, exhaustive validation lists, duplicate “current approach” sections, and detailed smoke-test commands for files that do not exist yet.

## Future Use
For similar module READMEs, use a compact contract summary plus one minimal module-call example when the input object is large. Group validation guardrails, identify stable downstream outputs, and keep live-provider testing docs to a short opt-in `real_tests/` note until the test and CI lane are actually implemented.
