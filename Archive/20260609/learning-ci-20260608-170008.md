# Learning: GitLab CI inspector docs need explicit pipeline source rules

## Context
During the multi-agent scoring sweep, the repo-local `gitlab-ci-pipeline-inspector` skill was reviewed against current GitLab documentation for merge request pipelines and predefined variable availability.

## Learning
GitLab CI investigation docs are most useful when they classify pipelines by `source`, especially treating `source == "merge_request_event"` as the authoritative MR readiness path when push pipelines are manual or different. The bridge/job/artifact/lint split is justified for dynamic child pipeline debugging, but repeated project IDs, hidden generator-job lookup assumptions, and destructive-looking shell examples reduce maintainability.

## Future Use
When writing or editing GitLab CI inspection guidance, define `PROJECT_ID` once, show an explicit MR pipeline filter by `source`, keep bridge inspection and generated-child-YAML linting, and include prerequisites such as `glab`, `jq`, authentication, and permissions. Use non-destructive wording for `.gitattributes` updates and label Linux-specific commands such as `sed -i` when portability matters.
